# Construire votre propre mAI

mAI peut être auto-hébergé sur votre propre infrastructure, vous offrant un contrôle total sur vos données, votre personnalisation et votre environnement de déploiement. Que vous déployiez pour une équipe, une organisation ou pour un usage personnel, mAI prend en charge plusieurs méthodes de déploiement.

<PlatformCards urlPrefix={'platform'} />

## Aperçu de l'architecture

mAI se compose de plusieurs composants clés :

### Services principaux

- **Application Next.js** — Frontend hybride SSR/SPA avec des routes d'API
- **Base de données PostgreSQL** — Stocke les conversations, les agents, les fichiers et les données utilisateur
- **Redis** (optionnel) — Stockage de session et mise en cache
- **Stockage compatible S3** — Téléversement de fichiers et documents de base de connaissances

### Services optionnels

- **RustFS / MinIO** — Stockage compatible S3 auto-hébergé
- **Langfuse** — Observabilité et traçage des LLM
- **OpenTelemetry** — Traçage distribué
- **Searxng** — Recherche web respectueuse de la vie privée

## Choisir une méthode de déploiement

### Docker Compose (Recommandé)

**Idéal pour :** Les installations auto-hébergées, le contrôle total de l'infrastructure, les déploiements d'équipe.

**Avantages :**

- Pile complète en une seule commande
- Mises à jour faciles avec `docker compose pull`
- Comprend PostgreSQL, Redis, RustFS, Searxng
- Prise en charge complète des fonctionnalités — pas de limite de temps, support de WebSocket

**Inconvénients :**

- Nécessite la gestion du serveur
- Nécessité de gérer les sauvegardes et la surveillance

### Vercel

**Idéal pour :** Les déploiements rapides, la mise à l'échelle sans serveur, une maintenance réduite.

**Avantages :**

- Déploiement en un clic
- HTTPS et CDN automatiques
- Mise à l'échelle automatique
- Offre gratuite disponible

**Inconvénients :**

- Nécessite une base de données PostgreSQL externe
- Limite de temps de fonction sans serveur de 10 secondes
- Pas de prise en charge des WebSockets
- Moins de contrôle sur l'infrastructure

### Plateformes cloud (Zeabur, Sealos, Dokploy)

Semblable à Vercel avec des options régionales. Idéal pour des exigences géographiques spécifiques avec diverses différences de prix et de fonctionnalités.

## Prérequis

Avant de déployer mAI, rassemblez les éléments suivants :

### Requis

**Clés API de fournisseurs d'IA** — Au minimum, vous avez besoin d'une clé API d'un fournisseur d'IA :

- **OpenAI** — `OPENAI_API_KEY` depuis [platform.openai.com](https://platform.openai.com/account/api-keys)
- **Anthropic** — `ANTHROPIC_API_KEY` depuis [console.anthropic.com](https://console.anthropic.com/)
- **Google** — `GOOGLE_API_KEY` depuis [aistudio.google.com](https://aistudio.google.com/app/apikey)

Consultez la [Configuration des fournisseurs d'IA](/docs/self-hosting/environment-variables/model-provider) pour obtenir la liste complète des fournisseurs pris en charge.

**Base de données (pour le déploiement serveur)** — PostgreSQL 14+ est requis :

- **Options managées** : Neon, Supabase, Railway, Vercel Postgres
- **Auto-hébergé** : Docker (inclus dans Docker Compose), AWS RDS, Google Cloud SQL

### Optionnel mais recommandé

**Redis** — Améliore les performances pour le stockage des sessions, la limitation du débit et la mise en cache. Utilisez Upstash, Redis Cloud ou Redis auto-hébergé.

**Stockage compatible S3** — Requis pour les téléversements de fichiers et les bases de connaissances :

- **AWS S3** — Prêt pour la production, évolutif
- **Cloudflare R2** — Pas de frais de transfert sortant
- **RustFS / MinIO** — Alternative S3 auto-hébergée (incluse dans Docker Compose)

**Alternative pour la compréhension visuelle** — Optionnel, mais recommandé si les utilisateurs téléversent des images ou des vidéos pendant la discussion avec des modèles qui n'ont pas de capacités visuelles natives. Configurez `VISUAL_UNDERSTANDING_PROVIDER` et `VISUAL_UNDERSTANDING_MODEL` avec un modèle doté de capacités visuelles de l'un de vos fournisseurs d'IA activés. Voir les [Variables d'environnement de base](/docs/self-hosting/environment-variables/basic#visual-understanding) pour plus de détails.

**Fournisseur d'authentification** — Pour l'authentification unique (SSO) et les fonctionnalités d'équipe (Google OAuth, GitHub OAuth, Microsoft Azure AD, Auth0, Keycloak). Voir la [Configuration de l'authentification](/docs/self-hosting/auth) pour la configuration.

## Considérations de sécurité

<Callout type={'warning'}>
  Ne committez jamais de clés API ou de secrets dans le contrôle de version. Utilisez toujours des variables d'environnement.
</Callout>

Mesures de sécurité essentielles :

1. **Utiliser HTTPS** — Déployez toujours avec des certificats SSL/TLS
2. **Sécuriser votre base de données** — Utilisez des mots de passe forts et limitez l'accès au réseau
3. **Variables d'environnement** — Stockez les secrets de manière sécurisée (jamais dans le code)
4. **Authentification** — Activez Better Auth pour les déploiements multi-utilisateurs
5. **Mises à jour régulières** — Gardez mAI et ses dépendances à jour

Options d'authentification :

- **Accès libre** — Aucune authentification (déploiements mono-utilisateur uniquement)
- **Better Auth** — Authentification intégrée avec e-mail/mot de passe, OAuth, liens magiques
- **Proxy inverse** — Utilisez Authelia, Authentik ou similaire

## Étapes suivantes

1. **Choisissez votre méthode de déploiement** — Docker pour un contrôle maximal ou Vercel for la simplicité
2. **Rassemblez les clés API** — Obtenez des clés API auprès des fournisseurs d'IA choisis
3. **Configurez l'infrastructure** — Approvisionnez la base de données, Redis et le stockage selon les besoins
4. **Configurez les variables d'environnement** — Voir la [référence des variables d'environnement](/docs/self-hosting/environment-variables)
5. **Déployez** — Suivez le guide spécifique à la plateforme ci-dessus
6. **Configurez l'authentification** — Configurez [Better Auth](/docs/self-hosting/auth) pour l'accès multi-utilisateur
